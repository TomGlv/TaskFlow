1. Introduction

This document describes the functional and non-functional requirements for TaskFlow, a web-based personal task management application. TaskFlow enables registered users to organise their work by creating, tracking, and categorising tasks.

1.1 Purpose

The purpose of this document is to provide a complete specification of the system behaviour so that developers, testers, and stakeholders share a common understanding of what the application must do.

1.2 Scope

TaskFlow covers: user account management, full task CRUD operations, task filtering and search, and category management. The initial release targets desktop and mobile web browsers.

1.3 Definitions

TermDefinition
TaskA unit of work with a title, description, status, priority, optional due date, and optional category.
CategoryA user-defined label used to group related tasks.
StatusThe current state of a task: To Do, In Progress, or Done.
PriorityThe urgency level of a task: Low, Medium, or High.
JWTJSON Web Token — a signed token used to authenticate API requests.
REST APIThe HTTP-based backend interface consumed by the frontend.

2. Functional Requirements

2.1 User Authentication

IDRequirementPriority
FR-01The system shall allow a visitor to register by providing a unique username, a unique email address, and a password of at least 6 characters.MUST
FR-02The system shall hash passwords using bcrypt before storing them in the database.MUST
FR-03The system shall allow a registered user to log in using their email address and password.MUST
FR-04Upon successful login or registration, the system shall issue a JWT valid for 24 hours.MUST
FR-05The system shall allow a logged-in user to log out, clearing the client-side session token.MUST
FR-06Protected API endpoints shall return HTTP 401 when no valid token is provided.MUST

2.2 Task Management

IDRequirementPriority
FR-07An authenticated user shall be able to create a task by providing at minimum a title. The system shall default status to To Do and priority to Medium.MUST
FR-08An authenticated user shall be able to view a list of all their tasks ordered by creation date descending.MUST
FR-09An authenticated user shall be able to update any field of their own task.MUST
FR-10An authenticated user shall be able to delete their own task.MUST
FR-11An authenticated user shall be able to mark a task as Done with a single click.MUST
FR-12The system shall prevent a user from accessing or modifying another user's tasks.MUST
FR-13An authenticated user shall be able to set an optional due date on a task.SHOULD
FR-14An authenticated user shall be able to add an optional description to a task.SHOULD

FR-16The system shall allow filtering tasks by priority (Low / Medium / High).MUST

IDRequirementPriority

3. Non-Functional Requirements

IDCategoryRequirement
NFR-01PerformanceAPI responses for standard operations (create, read, update, delete) must complete within 200 ms under normal single-user load.
NFR-02SecurityAll passwords must be stored as bcrypt hashes (cost factor ≥ 10). No plain-text credentials shall appear in logs or responses.
NFR-03SecurityJWT tokens must be signed with HS256 and must expire after 24 hours.
NFR-04UsabilityThe user interface must be usable on screen widths from 360 px (mobile) to 1920 px (desktop) without horizontal scrolling.
NFR-05ReliabilityThe application must sustain 99 % uptime during normal operation on a standard server (excluding scheduled maintenance).
NFR-06MaintainabilityThe codebase must be organised into separate backend (Node.js/Express) and frontend (HTML/CSS/JS) modules. Each backend route must reside in its own file.
NFR-07Data IntegrityThe database must enforce foreign key constraints. Cascading deletes must remove orphaned task and category records when a user account is deleted.
NFR-08TestabilityAll critical API endpoints must have automated unit/integration tests with at least 80 % code coverage on business logic.
NFR-09PortabilityThe application must run on Windows, macOS, and Linux using Node.js ≥ 18.

4. Assumptions, Concepts and Constraints

4.1 Assumptions

  • Each task belongs to exactly one user; there is no task sharing between accounts in the initial release.
  • Users have a reliable internet connection; the application does not need offline support.
  • Users access the application through a modern web browser with JavaScript enabled (Chrome ≥ 90, Firefox ≥ 88, Edge ≥ 90, Safari ≥ 14).
  • The server environment has Node.js 18+ installed and write access to the filesystem for the SQLite database file.
  • The initial user base is small (fewer than 1 000 concurrent users), so SQLite is an appropriate database choice.

4.2 Conceptual Model

The core domain model is described below:

EntityAttributesRelationships
Userid, username, email, password (hashed), created_atHas many Tasks; has many Categories
Taskid, title, description, status, priority, due_date, created_at, updated_atBelongs to User; optionally belongs to one Category
Categoryid, nameBelongs to User; has many Tasks

4.3 Constraints

  • Password length: Minimum 6 characters enforced at both client and server level.
  • Task title: Mandatory; cannot be blank.
  • Task status values: Restricted to todo, in_progress, done (enforced by a database CHECK constraint).
  • Task priority values: Restricted to low, medium, high (enforced by a database CHECK constraint).
  • Token expiry: JWT tokens are valid for 24 hours; clients must re-authenticate upon expiry.
  • Category name: Cannot be empty or contain only whitespace.
  • Database: SQLite file stored locally on the server; no external database server is required.
  • Unique credentials: Username and email address must each be unique across all users.

4.4 Out of Scope

  • Email verification on registration
  • Password reset via email
  • Real-time collaboration / task sharing
  • File attachments on tasks
  • Push notifications
  • Mobile native applications (iOS/Android)
